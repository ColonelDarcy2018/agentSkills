# 手机 RPA 脚本开发标准工作流

## 1. 前置检查

1. 确认目标信息
- 目标 App（名称/包名）
- 目标页面与动作
- 成功判定（看到什么日志/页面变化）

2. 确认执行环境
- 调用 `list_connected_devices`
- 若为空：调用 `list_adb_devices` 并引导 `connect_device`
- 若 `tars_connected=false`：先修复塔斯可达性（启动 App、检查端口转发）

3. 设备选择策略
- 只有 1 台设备：自动使用
- 多台设备：先让用户明确选择 `device_serial`

## 2. 脚本生成策略

先输出最小可运行脚本，再做增强：

1. 入口函数固定
```python
def main(zbot, *args, **kwargs):
    ...
```

2. 最小主路径
- 启动应用
- 等待稳定
- 1~2 个关键动作
- 输出结果日志

3. 可观测性约束
- 每个关键步骤前后都使用 `zbot.log()`
- 必要时加 `zbot.sleep()` 防止竞态

## 3. 执行与观测

1. 调用 `execute_rpa_code`，记录 `job_id`
2. 轮询 `get_execution_logs(job_id, device_serial, lines)`
3. 需要视觉证据时调用 `take_screenshot`

建议轮询节奏：
- 初始阶段：1~2 秒一轮
- 稳态阶段：2~3 秒一轮

## 4. 修复迭代

每轮只修一类根因：

1. 设备/连接问题
2. 权限/环境问题
3. 元素定位问题
4. 点击动作问题
5. 语法/结构问题

修复后立即重跑，避免叠加改动。

## 5. 交付规范

最终回答必须包含：

1. 环境状态结论
2. 最终可执行脚本
3. 执行证据（job_id、关键日志）
4. 下一步建议（增强点或人工介入点）
