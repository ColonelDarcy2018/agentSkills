# Game Requirement Innovation

## 作用
- 在游戏立项期先完成需求基线对齐，再完成玩法创新收敛。
- 产出可直接衔接后续规划与实现的结论文档。

## 触发边界

### 触发
- 用户明确处于“需求模糊、目标未定、边界未定”的立项早期。
- 用户要求“玩法创新、机制融合、先发散后收敛”。

### 不触发
- 需求已冻结且仅要求直接编码或纯排障。

## 协议引用
- 公共规则：`KCOS/protocol/p0-rules.md`
- 会话清单：`KCOS/protocol/ai-playbook.md`
- 输出模板：`references/requirement-understanding-template.md`
- 提问清单：`references/question-bank.md`
- 创新流程：`references/innovation-diverge-converge.md`

## 最小流程
1. 需求基线澄清：明确目标、用户、场景、范围、约束、验收。
2. 创新目标定义：明确创新目的与创新强度。
3. 发散创意池：按核心循环、成长、对抗、关卡/模式、元玩法产出至少 `8` 个候选。
4. 创意收敛评分：按 `新颖性/可实现性/乐趣潜力/MVP 适配度/风险可控性` 评分排序。
5. 决策切片：确定 `1` 个稳定基线方案 + `1` 个创新主方案 + `1` 个备选方案。
6. 验证计划：为创新主方案定义假设、最小实验、指标与终止条件。
7. 输出需求理解结论：按模板沉淀并请求用户确认关键决策点。

## 输出契约
- 必含：`背景目标 + 玩家场景 + 核心体验与范围 + 约束与验收 + 创新候选 + 收敛评分 + 方案决策 + 验证计划 + 待拍板问题`。
- 信息不足时先给“假设版结论 + 3-5 个关键问题”。

## 执行规则
- 优先澄清“做什么、为什么、如何验收”，不提前陷入实现细节。
- 单轮提问默认 `3-5` 个，优先能改变方案路径的问题。
- 显式标注事实、假设、推断，禁止把假设写成既定事实。
- 创新方案必须给出“保留理由 + 淘汰理由”，避免拍脑袋决策。
- 收敛时必须保留可落地基线，避免全盘高风险方案。
- 如新增/修改知识文档，结束前执行 `python3 KCOS/scripts/kcos_p0.py sync`。

## 手动触发示例
- "先做游戏需求理解，再做创新设计"
- "我想法很散，先发散再收敛"
- "给我基线方案和创新方案，再帮我选 MVP"
